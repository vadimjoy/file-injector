## В Одном Предложении

`ai-css-kit` — это детерминированная визуальная платформа для быстрой и точной сборки интерфейсов из ограниченного набора CSS-примитивов, тем, схем и переиспользуемых контрактов компонентов.

## Какую Проблему Решает Проект

Большинство AI- или low-cost-UI workflows ломаются двумя способами:

1. генерируют сырую разметку и CSS, которые быстро расползаются;
2. опираются на слишком общий UI kit, который не гарантирует визуальную точность.

`ai-css-kit` идёт в противоположную сторону:

- один раз описать контракт компонента;
- один раз задать допустимые вариации;
- вынести их в схемы и токены;
- собирать интерфейсы из этих ограничений, а не изобретать UI заново.

## Ментальная Модель

Проект нужно читать так:

```text
CSS kit -> themes -> showcase/playground -> CLI
```

### 1. CSS kit

Канонический слой:

- токены
- foundations
- utilities
- компоненты

Именно здесь начинается визуальная точность.

### 2. Theme system

Слой, который превращает наборы токенов и JSON-описания тем в реальные CSS-темы.

### 3. Showcase / playground

Визуальная лаборатория и контрольный стенд.

Именно здесь компонент должен доказать, что выдерживает:

- вариации размеров
- смену состояний
- token overrides
- переключение тем
- будущую интеграцию с валидатором и CLI

Если playground нестабилен, визуальный контракт тоже нестабилен.

### 4. CLI

Будущий верхний слой.

Он не должен изобретать CSS с нуля. Он должен собирать валидный HTML из уже стабильных контрактов, схем и правил резолвера.

## Чем Проект Не Является

- не свободным визуальным конструктором
- не LLM, которая пишет произвольный CSS с нуля
- не просто витриной демо
- не проектом “в первую очередь про CLI”

## Текущее Состояние

### Уже готово

- CSS kit
- токены и темы
- audit/lint скрипты
- направление компонентных контрактов

### Активно стабилизируется

- `showcase-app`
- schema-driven playground
- покрытие компонентов схемами

### Зависит от зрелости playground

- CLI end-to-end поток

Это важно: playground — не побочный бонус, а блокер для CLI.

## Куда Безопасно Контрибьютить

Если человек только входит в проект, лучше начинать здесь:

1. токены и theme mapping
2. utilities, grid и layout primitives
3. cleanup CSS-контрактов существующих компонентов
4. schema-driven playground для существующих компонентов
5. документация и onboarding

## Короткий Ответ Для Друга

Если кто-то спрашивает “что это за проект?”, отвечай так:

> Это constrained visual system для быстрой и точной сборки интерфейсов.
