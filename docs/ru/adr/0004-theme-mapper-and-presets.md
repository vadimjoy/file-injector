# ADR-0004 — Theme Mapper, каскадные слои и библиотека пресетов

> **Язык:** Русский | [English](../../en/adr/0004-theme-mapper-and-presets.md)

**Статус:** Accepted  
**Дата:** 2026-04-07  
**Автор:** vadimjoy  
**Связанные ADR:** [0003](0003-ai-token-naming.md)

---

## Контекст

Фаза 3 дорожной карты требовала придать темизации такую же детерминированность, как и атомам из Фазы 1/2. На версии v0.5.0 в проекте:

- Существовал только `:root` (light) и `[data-theme="dark"]` override внутри `tokens.css` — без формального формата для сторонних тем.
- AI-агенты и дизайнеры вынуждены вручную сопоставлять внешние токены с `--ui-*` / `--ai-*`, что занимало время и часто приводило к ошибкам.
- Компонентные стили не были заключены в `@layer`, поэтому итоговая специфичность зависела от порядка импорта; любые темы приходилось подключать «после всего» и бороться со стилями через `!important`.
- Демо-страницы и showcase переключали только бинарный `light/dark`, не демонстрируя новые пресеты.

Нужно было описать машиночитаемый формат тем, автоматизировать генерацию CSS и гарантировать предсказуемость каскада.

---

## Решение

1. **Каскадные слои** — в `src/css/index.css` объявлен порядок `@layer ai-kit.tokens, ai-kit.base, ai-kit.components, ai-kit.themes`. Все существующие файлы (`tokens.css`, `base.css`, компонентные CSS, `states.css`) обёрнуты в соответствующие слои, чтобы темы в `ai-kit.themes` всегда побеждали без `!important`.

2. **Theme Mapper CLI** — создан `scripts/theme-map.js`, который:
   - Принимает JSON темы в трёх форматах: родной (`meta` + плоские `tokens`), W3C Design Tokens (`$value`/`$type`) и Figma Tokens (`value`/`type`).
   - Автоматически распознаёт формат, разворачивает дерево в dot-notation и сопоставляет ключи с `--ui-*` и `--ai-*` переменными.
   - Генерирует CSS-файл с комментариями и `@layer ai-kit.themes { [data-theme="name"] { ... } }`.

3. **Сборка пресетов** — добавлены исходные JSON-файлы в `src/themes/` (default, dark, midnight, corporate, warm) и скрипт `scripts/build-themes.js` + npm-скрипт `build:themes`, который транслирует их в `dist/themes/*.css` и публикует через `package.json#exports./themes/*`.

4. **Документация** — написан `docs/theming.md`, обновлены `README.md`, `SPEC.md`, `AI_CONTEXT.md`, `MASTER_PLAN.md` и release notes с описанием формата, CLI и новых кнопок в демо.

5. **Демо-витрина** — `index.html` теперь имеет селектор пресетов (default/dark/midnight/corporate/warm). `showcase-page.js` рассылает выбранную тему во все iframe, а `demo-theme.js` применяет любое значение `data-theme` (не только dark). Все демо импортируют themes CSS, чтобы реагировать на этот выбор.

## Последствия

### Положительные

- Темы стали детерминированными: любой JSON → CLI → готовый CSS, который гарантированно перекрывает компоненты без ручного тюнинга специфичности.
- Поддержка W3C/Figma форматов избавляет дизайнеров и AI-агентов от ручной перекодировки токенов.
- Появились официальные пресеты, которые можно подключить строкой `import 'ai-css-kit/themes/midnight';`, что снижает порог входа.
- Демо повторяют продакшен-поток: `data-theme="midnight"` и набор файлов `dist/themes/*.css` тестируются в реальном UI.
- Тесты (`node --test tests/theme-map.test.js`) обеспечивают регресс на всю цепочку парсинг → генерация.

### Отрицательные / Компромиссы

- Сборка стала длиннее: помимо `build:css` теперь всегда запускается `build:themes`.
- Бандл демо-страниц грузит дополнительные CSS-файлы (все 5 пресетов), чтобы можно было мгновенно переключаться — увеличенный вес приемлем для showcase, но не для потребителей (в prod импортируют только нужные темы).
- CLI и тесты требуют Node ≥18 из-за использования встроенного `node:test` и современных API.
- Нужно поддерживать JSON-спецификацию и синхронизировать её с документацией (дополнительный процесс при добавлении новых токенов).

---

## Альтернативы, которые рассматривались

1. **Оставить только `:root` + `[data-theme="dark"]`** — просто задокументировать, как переопределять `--ui-*`. Отклонено: не решает задачу быстрого импорта чужих дизайн-систем, не даёт presets, не масштабируется для AI-агентов.

2. **Использовать внешние инструменты (Style Dictionary / Theo)** — требовало бы тяжёлого Node-пайплайна и JSON-DSL, не дружелюбного к AI. Наша цель — минимальный вход: плоские ключи, очевидные правила именования, zero-dependency CLI.

3. **Генерировать темы через PostCSS на лету** — можно было бы импортировать JSON в PostCSS, но это привязывает пользователей к нашему build-стеку. CLI, основанный на `node scripts/theme-map.js`, позволяет использовать инструмент как внешнюю утилиту и в CI, и локально.

4. **`prefers-color-scheme` вместо `data-theme`** — решает только бинарный dark/light сценарий, без именованных пресетов и без поддержки дизайнерских JSON.

---

## Релиз / Follow-up

- Включить `npm run build:themes` в основной пайплайн `npm run build` (выполнено).
- Поддерживать индекс тем в документации и демо.
- При добавлении новых токенов обновлять `src/themes/*.json`, `docs/theming.md` и unit-тесты Theme Mapper.
